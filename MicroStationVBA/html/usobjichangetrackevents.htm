IChangeTrackEvents Object

                                                     

Lets programs monitor all of the changes that happen to DesignFiles during a MicroStation session. It lets programs react to those changes. This is useful, for example, where the program keeps redundant, in-memory information that is somehow correlated with the current state of the elements in a DesignFile.

Importantly, the Change Track API is NOT intended to enable programs to "influence" or modify the changes to the DesignFile. Obviously, many programs can monitor the Change Track events simultaneously, and the order in which they are processed is indeterminate. However, since none of them can change the information being processed, the order of execution is moot.

There are two types of changes that happen to a DesignFile during a MicroStation session: direct changes, and undo/redo changes. Direct changes are just that - they directly modify the state of the DesignFile. Direct changes happen either as a result of input from the user, or by calls from a program. The DesignFile can be viewed as a database that is driven from one consistent state to another consistent state via a series of direct changes. The Change Tracking listeners are notified as each change is made to the DesignFile.

Through the UNDO buffer, MicroStation keeps track of groups of changes that together constitute a consistent set of changes, and provides a user interface to reverse unwanted changes. A "mark" delineates the end of a logically related group of changes. The undo command reverses all changes between two marks, and the redo command restores all changes between two marks. The Change Track listeners are notified as marks are created and as files are modified by undo's and redo's.

Change Track distinguishes between direct changes and undo/redo changes to allow programs to "propagate" further changes. Change propagation can happen recursively (that is, one change can cause another change that causes still other changes) but only as a result of direct changes. Change propagation cannot happen when reversing changes via undo/redo because the file must return to a previous known-to-be-valid state.
